## 1.2 Gate定义

### Gate-1: 启动期三段式机制可行性

| 项 | 内容 |
|----|------|
| 验证 | 三段式（播种-验证-加码）能否有效保障用户体验同时给主播机会 |
| MVP | MVP-0.1, MVP-0.2 |
| 若通过 | 留存稳定 + 负反馈可控 → 上线三段式机制 |
| 若失败 | 体验受损严重 → 调整门槛或回滚 |
| 状态 | ⏳ |

### Gate-2: 高潜主播识别模型效度

| 项 | 内容 |
|----|------|
| 验证 | 潜力评分卡能否有效预测30/60天主播LTV或生存概率 |
| MVP | MVP-1.1, MVP-1.2, MVP-1.3 |
| 若AUC>0.7 | 上线分层探索预算 |
| 若AUC<0.6 | 需增加特征或调整标签定义 |
| 状态 | ⏳ |

### Gate-3: 用户意图分层排序有效性

| 项 | 内容 |
|----|------|
| 验证 | 差异化排序权重是否提升用户体验 + 转化效率 |
| MVP | MVP-2.1, MVP-2.2 |
| 若通过 | 上线用户分层策略 |
| 若失败 | 简化为粗分层或放弃 |
| 状态 | ⏳ |

### Gate-4: 成长信号产品化对留存的因果效应

| 项 | 内容 |
|----|------|
| 验证 | 成长信号能否提升主播开播留存 |
| MVP | MVP-3.1, MVP-3.2 |
| 若留存>10% | 优先级提升，全量产品化 |
| 若留存<5% | 仅保留基础功能 |
| 状态 | ⏳ |

# 3. MVP规格

## Phase 0: 基础建设

### MVP-0.1: KPI指标字典

| 项 | 配置 |
|----|------|
| 目标 | 定义完整的KPI体系（主/分解/护栏），精确到分子分母 |
| 主KPI | HP Supply增长、HP 30天生存率 |
| 分解KPI | 拉新漏斗、启动期质量、成长结果 |
| 护栏KPI | 用户侧（负反馈/时长/回访）、生态侧（头部指标/多样性） |
| 口径 | 有效开播=7天>=2场且>=60min；高潜=生存+体验+成长 |
| 验收 | 指标字典文档 |

### MVP-0.2: 启动期门槛数据分析

| 项 | 配置 |
|----|------|
| 目标 | 确定三段式机制各阶段的门槛数值（按品类分桶） |
| 数据 | 历史直播日志（近90天） |
| 分析维度 | 品类 x 语言 x 地区 |
| 关键指标 | 首分钟留存、评论/分钟、负反馈率 |
| 验收 | 各品类门槛建议值 + 置信区间 |

## Phase 1: 高潜主播识别

### MVP-1.1: 主播特征工程（开播前）

| 项 | 配置 |
|----|------|
| 目标 | 构建开播前可用的主播特征集 |
| 特征类别 | 账号历史、画像、可持续性线索 |
| 验收 | 特征覆盖率 + 特征重要性分析 |

### MVP-1.2: 主播特征工程（启动期）

| 项 | 配置 |
|----|------|
| 目标 | 构建开播后5-10分钟可用的启动期特征 |
| 特征类别 | 留存曲线、互动质量、内容健康度、转化漏斗 |
| 验收 | 特征工程完成 + 探索性分析 |

### MVP-1.3: 高潜主播预测模型

| 项 | 配置 |
|----|------|
| 目标 | 预测主播30/60天LTV或生存概率 |
| 标签 | 30天LTV或30天生存指标 |
| 模型 | GBDT + 生存分析 |
| 验收 | AUC>0.7, 可解释性报告 |

## Phase 2: 用户意图分层

### MVP-2.1: 用户意图分层定义

| 项 | 配置 |
|----|------|
| 目标 | 建立轻量可解释的用户分层 |
| 分层 | 付费高意图、社交互动型、内容消费型、新用户/冷启动 |
| 验收 | 分层覆盖率 + 分层区分度分析 |

### MVP-2.2: 差异化排序权重实验

| 项 | 配置 |
|----|------|
| 目标 | 验证不同人群差异化排序权重的效果 |
| 实验设计 | Bandit或分桶A/B |
| 验收 | 体验提升>5%且无显著负面影响 |

## Phase 3: 成长信号产品化

### MVP-3.1: 成长信号产品设计

| 项 | 配置 |
|----|------|
| 目标 | 设计可见、可追踪、可达成的成长信号体系 |
| 内容 | 成长仪表盘、里程碑任务、社区认同、流量可解释 |
| 验收 | 产品PRD + 原型 |

### MVP-3.2: 成长信号A/B实验

| 项 | 配置 |
|----|------|
| 目标 | 验证成长信号对主播开播留存的因果效应 |
| 实验设计 | 随机分流A/B（同批高潜候选） |
| 验收 | 留存提升>10%且统计显著 |

# 4. 模块架构与 IO

链路：召回 -> 排序/出牌 -> 金牌/加码

## 4.1 模块 1：高潜主播识别（Offline/准实时）

### Inputs

| 类别 | 字段 |
|------|------|
| 主播静态 | 账号信息、历史开播、短视频表现、内容品类/语言/地域、合规风险信号 |
| 直播间表现 | 首1/3/5分钟留存、评论/分钟、关注率、负反馈 |
| 其他模型 | 内容embedding、用户兴趣embedding、基础排序分 |

### Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| anchor_potential_score | float [0,1] | 高潜概率/分数 |
| anchor_tier | enum | S/A/B/C 分层 |
| reason_codes | list[str] | 可解释原因 |
| actionable_list | list | 给运营的名单 |

---

## 4.2 模块 2：开播启动期流量分配（Online，分钟级决策）

### Inputs

| 类别 | 字段 |
|------|------|
| 主播分层 | tier/score（来自模块1） |
| 实时信号 | 当前UV、并发、留存斜率、评论/分钟、关注率、负反馈率 |
| 预算/约束 | 新主播探索预算、单场最大曝光上限、负反馈止损线 |

### Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| seed_quota | int | 启动播种曝光量 |
| scale_policy | dict | 加码规则 |
| stop_rule | dict | 止损条件 |
| traffic_multiplier | float | 给排序层的加权系数 |

---

## 4.3 模块 3：人群匹配

### Inputs

| 类别 | 字段 |
|------|------|
| 用户侧 | 兴趣embedding、语言/地域、Live偏好、付费/互动意图分层、新用户标识 |
| 主播侧 | 内容品类、历史吸引人群、实时房间氛围 |
| 约束 | 新用户保护、低付费用户保护、合规风险 |

### Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| eligible_user_cohorts | list | 适配人群池（白名单） |
| exclude_user_cohorts | list | 不适配人群（黑名单/降权） |
| match_score | float | 用户-直播间匹配分 |

---

## 4.4 模块 4：运营协同交付

### Inputs

| 来源 | 内容 |
|------|------|
| 模块1 | 高潜名单 |
| 模块2 | 启动期失败原因 |
| 模块3 | 匹配偏差诊断 |

### Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| 运营可执行包 | 周更/日更 | 重点跟进主播名单 |
| 建议动作 | list | 每个主播1-3条建议 |
| 原因分类 | enum | 为什么拿不到量 |
| 反馈回流字段 | dict | 是否触达、采纳、效果 |

---

## 4.5 模块接口规格速查

| 模块 | 刷新频率 | 延迟要求 | 存储 |
|------|---------|---------|------|
| 模块1：高潜识别 | 日更 + 分钟级 | 离线T+1，在线<1s | 离线表 + Redis |
| 模块2：流量分配 | 分钟级 | <100ms | 在线服务 |
| 模块3：人群匹配 | 实时 | <50ms | 在线服务 |
| 模块4：运营交付 | 日/周更 | T+1 | 离线表 + Dashboard |
